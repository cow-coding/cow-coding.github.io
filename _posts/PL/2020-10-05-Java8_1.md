# JAVA 활용

자바는 우리나라에서 많이 쓰는 프로그래밍 언어 중 하나이다. jdk14 버전까지 배포가 되었으면 가장 많이 쓰이는 버전은 대체적으로 jdk8 버전이다.  
jdk가 발전하면서 새로운 기능들을 추가하였다.  
그 대표적인 예시가 JAVA 8부터 추가된 Stream과 lambda식이다.

# 프로그래밍 방식

프로그래밍 언어는 크게 3가지 분류로 나눠지게 된다.

1. 절차지향 언어
2. 객체지향 언어
3. 함수형 언어

- 일반적으로 1번과 2번을 '명령형 프로그래밍'이라고 부른다. 일반적으로 C, C++, Java와 같은 언어들이 이에 해당된다. 우리가 가장 익숙하게 받아들이는 프로그래밍 방식이다.  
이 방식은 대체적으로 코딩방식이 "어떤 방법으로 하는 지 (How)"을 하고 싶은 지를 보여준다.

- 3번과 같은 프로그래밍 방식을 '선언형 프로그래밍'이라고 부른다. 대표적인 함수형 프로그래밍 언어는 하스켈, 리스프 같은 언어가 있다. 이런 방식은 명령형 프로그래밍을 하던 사람이 접하면 굉장히 낯설게 느껴진다.  
그 이유는 간단한데, 이 방식의 코딩방식은 "무엇 (What)"에 집중하여 작성하기 때문이다.

어떻게와 무엇 이라고 말하니까 생각보다 와 닿지가 않는다. 간단하게 예를 들어서 이야기를 해보겠다.  

> **"여러분은 지금 식당에 와 있다. 앉을 자리를 정하기 위해 직원에게 뭐라고 해야할까?"**

- 명령형 방식 (How) : "(창가 옆 )1번 자리가 비었네요. 저는 1번 자리로 걸어가서 앉을 거에요."
- 선언형 방식 (What) : "1명 앉을 자리 부탁드려요."

이렇게 보면 대충 뭔 차인지가 눈에 보일 것이다. 명령형 방식은 말 그대로 자리에 앉는 방법에 관점을 두고 있다. 하지만 선언형 방식은 내가 원하는 것을 전달하는 것에 중점을 두고 있다.  
위에서 명령형 프로그래밍에 익숙한 사람들이 선언형 프로그래밍을 접하면 굉장히 낯설게 느껴지는 이유가 여기서 나타난다.

<center> <b>"그렇다면 나는 자리를 어떻게 이동하는거지??"</b> </center><br>

선언형 프로그래밍이 동작하기 위해서는 명령형 방식으로 중간 매개체가 추상화 되어 있어야 한다.   
우리가 식당에서 가서 "1명 앉을 자리 부탁해요"하고 자리로 어떻게 가지?라고 고민하지 않고 직원에게 맡기는 것처럼 말이다.  

이에 대한 자세한 내용은 후에 프로그래밍 언어의 패러다임에서 다루겠다.

# 람다( $$\lambda$$ )표현식과 함수형 프로그래밍

Java는 8부터, C++은 11부터 공식적으로 람다식을 지원한다. 람다식이 나타나면서 간결하고 보기가 편한 코드작성이 가능하게 되었다. 람다 표현식을 사용하면서 다음과 같은 장점을 갖게되었다.

1. 함수적 프로그래밍이 가능하다.
2. 가독성 좋고 명확한 프로그래밍이 가능하다.
3. API나 라이브러리 사용에 용이하다.
4. 병렬처리 연산에 유용하다.

람다 표현식의 기본 작성 방식은 아래와 같다.

```java
public static void main(String[] args) {
  BigNumber max = (a, b) -> (a >= b) ? a : b;
  // (매개변수 목록) ->  { 실행구문; 실행구문; }
}
```

이제 기존 작성방식과 람다식 활용 방식을 비교해보자.

< 기존 작성 방식 >

```java
public class Test{
  public static void main(String[] args) {
    int max;
    int a = 1, b = 5;

    if (a >= b) max = a;
    else max = b;

    System.out.println(max);
  }
}
```

< 람다식 활용 방식 >

```java
public class Test {
  @FunctionalInterface

  public interface BigNumber{
    int getMax(int n1, int n2);
    // 추상 메서드가 1개인 경우만 함수형 인터페이스라고 한다.
  }

  public static void main(String[] args) {
    BigNumber max = (a, b) -> (a >= b) ? a : b;

    System.out.println(max.getMax(1, 5));
  }
}
```

사실 부피가 작은 코딩이어서 크게 차이가 있어 보이지는 않는다. 하지만 부피가 커지면 이런 람다 표현식을 상당히 큰 힘을 갖게된다. 후에 서술하는 Stream과 함께 사용하면 더 효과를 볼 수 있다.  
람다 표현식을 활용한 함수형 프로그래밍은 자세히 들여다보면 특이한 점을 알 수 있다. max라는 변수에 getMax라는 메서드가 있고 호출이 가능한 것은 알겠는데... max에 값을 어떻게 전달한 것이지?라는 생각이 들 수 있다.  
함수형 프로그래밍의 구조를 간단하게 설명을 해보겠다.

![](/image/pl/lambdaex.png)

BigNumber라는 interface에 max는 선언이 되는 동시에 람다식의 메모리 주소가 할당이 되게 된다. 즉, max는 마치 어떤 값을 받는 것처럼 보이지만 사실 함수 주소를 담고 있는 변수가 되게 된다.
max가 호출되면서 기본적인 람다식 표현법에 따라서 max는 순서대로 인자값을 받아온다. 그리고 getMax가 호출될 때, max는 저장한 람다식 코드 자체를 paramter로 getMax에게 넘겨준다.  
이렇게 interface를 사용하다보면 인자 자료형 불일치 오류나 추상 메서드를 2개 이상 쓰거나 하는 오류가 나타날 수 있다.  
이때, 문제가 일어나는 interface를 찾기 편하게하고 이러한 오류를 방지하기위해서 java는 Functional Interface annotation(`@FunctionalInterface`)를 작성하는 것을 지원한다.

# Java Stream

Stream도 java 8 버전부터 지원하기 시작한 기능이다. 전체적으로 데이터를 한 흐름에 처리하는 방식이다. 스트림과 람다식을 활용하게 되면 방대하게 긴 코드 로직을 한 흐름에 간단하게 짤 수 있다.  
또한 parallel stream을 활용해서 병렬처리에서도 큰 효과를 볼 수 있다.  
자바 스트림은 크게 3가지 과정으로 이루어져 있는데, terminal operation이 나오기 전까지는 intermediate operation은 실행되지 않는다.

1. 생성
2. 가공 (intermediate operation)
3. 결과 (terminal operation)

피보나치 함수를 일반적인 방식과 자바 스트림 방식으로 비교해보자.

< 일반적인 방식 >
```java
import java.util.Scanner;
import java.util.stream.Stream;

public class Fibonacci {

  public static int fibo(int n) {
    if (n <= 1) return n;
    else fibo(n - 1) + fibo(n - 2);
  }

  public static void main(String[] args) {
    Scanner sc = new Scanner(System.in);
    System.out.print("정수 입력 : ");
    int j = sc.nextInt();

    System.out.println(fibo(j));
  }
}
```

< Java Stream 방식 >
```java
import java.util.Scanner;
import java.util.stream.Stream;
import java.math.BigInteger;

public class Fibonacci {
  public static void main(String[] args) {
    Scanner sc = new Scanner(System.in);
    System.out.print("정수 입력 : ");
    int j = sc.nextInt();

    Stream.iterate(new int[] {1, 1}, p -> new int[]{p[1], p[0] + p[1]})
    .limit(j)
    .forEach(p -> System.out.println(p[0]))
  }
}
```

Stream을 사용해서 하나의 흐름으로 코드를 작성한 것을 볼 수 있다. 둘이 조금 다르기는 한데, 살짝 수정하면 어느정도 비슷하게 만들 수는 있다.  
iterate, limit은 가공과정, forEach는 결과과정이라고 보면된다.
